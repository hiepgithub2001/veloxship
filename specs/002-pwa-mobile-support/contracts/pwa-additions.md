# Phase 1 — Contracts: PWA Additions

This feature adds **no new HTTP endpoints**. The contract surface that *is* introduced is:

1. The web app manifest (a static contract between the app and the browser/OS).
2. The service worker registration (a contract between the page and the browser).
3. Reused HTTP endpoints from `001-logistics-bill-app` (documented here for cross-reference; the contracts themselves are unchanged).

## 2. Service worker registration contract

The Vite-PWA plugin generates `frontend/dist/sw.js` from a Workbox manifest. The page registers it via `virtual:pwa-register`:

```js
// frontend/src/pwa/registerSW.js
import { registerSW } from 'virtual:pwa-register';

export const updateSW = registerSW({
  immediate: true,
  onNeedRefresh() {
    // Surfaced to <UpdatePrompt> via a callback / event bus.
    // The component shows the Vietnamese "Có phiên bản mới — tải lại để cập nhật" affordance.
    // It is suppressed if the URL matches /bills/new/mobile (mid-wizard guard).
  },
  onOfflineReady() {
    // No-op — we are online-only. The plugin still calls this once on first install
    // because the app shell is precached, but the user is not told the app "works offline".
  },
});
```

**Workbox configuration** (in `frontend/vite.config.js`'s `VitePWA` block):

```js
VitePWA({
  registerType: 'prompt',
  injectRegister: false, // we register manually in main.jsx
  workbox: {
    globPatterns: ['**/*.{js,css,html,svg,png,woff2}'],
    navigateFallback: '/index.html',
    navigateFallbackDenylist: [/^\/api\//],
    runtimeCaching: [
      {
        urlPattern: /^\/api\//,
        handler: 'NetworkOnly',
      },
    ],
    cleanupOutdatedCaches: true,
  },
  manifest: { /* see §1 */ },
})
```

**Acceptance**:
- FR-009: `NetworkOnly` for `/api/*` prevents any business-data caching.
- FR-013, FR-014: `registerType: 'prompt'` + `<UpdatePrompt>` mid-wizard suppression deliver auto-update without interruption.

## 3. Reused HTTP endpoints (no contract change)

The mobile UI consumes the existing 001 endpoints. Listed here for traceability; see `specs/001-logistics-bill-app/contracts/openapi.yaml` for full schemas.

| Endpoint | Used by mobile flow | Spec FR / scenario |
|----------|---------------------|--------------------|
| `POST /api/auth/login` | Mobile login screen | FR-004 |
| `POST /api/auth/logout` | Mobile avatar menu | FR-017, FR-020 |
| `GET /api/bills?...` | Mobile lookup screen (search by tracking number, phone) | FR-005, US1-AC3 |
| `GET /api/bills/{id}` | Mobile bill detail | FR-005, US1-AC3 |
| `POST /api/bills` | Mobile sectioned create-bill wizard (final submit from `<ConfirmStep>`) | FR-005, FR-008a..d, US1-AC6 / US1-AC6a |
| `PATCH /api/bills/{id}/status` | Mobile status-update screen | FR-005, US1-AC3 |
| `GET /api/bills/{id}/print?format=pdf` | Mobile PDF preview screen post-create | FR-008d, US1-AC6a |
| `GET /api/customers?...` | Mobile customer lookup / sender auto-fill | FR-005 |

**No new endpoints. No request/response schema changes. No new auth scopes.**

## 4. What is explicitly NOT in the contract surface

- **No** `/api/system/version` endpoint — the version label is a build-time constant baked into the bundle (research.md §9, data-model.md `App Version Record`).
- **No** push subscription endpoints (`POST /api/push/subscribe`, etc.) — push notifications deferred (clarification Q3).
- **No** offline sync endpoints (`POST /api/sync/...`) — offline removed (clarification Q1).
- **No** scanner upload endpoints — barcode scanning deferred (clarification Q5).

These absences are deliberate and preserve the "no new domain capabilities" stance of the plan.
